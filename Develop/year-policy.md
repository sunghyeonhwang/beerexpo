# Year Policy (연도 표기 정책)

- Project: beer_EXPO (KIBEX 2026)
- Updated: 2026-02-20
- Scope: 프로젝트 전반의 연도 표기 및 과거 데이터 처리 규칙

## 1) 기본 원칙

### 현재 행사 연도: 2026
- 모든 페이지의 주요 일정/장소/참가 안내는 **2026년 기준**으로 작성
- Hero 섹션, CTA, 메인 문구는 2026 명시

### 전년도 실적: 2025
- 참가사 수, 방문객 수, 프로그램 운영 실적 등은 **2025년 실제 데이터** 기준
- 반드시 `전년도 기준` 또는 `2025년 기준` 라벨 병기

### 과거 데이터: 2024 이전
- 아카이브 성격의 데이터만 표기
- `아카이브` 또는 구체 연도 명시 (예: `2024년 성과`)

## 2) 섹션별 표기 규칙

### Hero / 메인 CTA
```
✅ 올바른 예시:
"KIBEX 2026 | 2026년 5월 14-16일 | 코엑스"
"2026 대한민국 맥주 박람회에 참가하세요"

❌ 잘못된 예시:
"KIBEX | 연례 행사" (연도 누락)
"매년 5월 개최" (구체성 부족)
```

### 실적/성과 데이터
```
✅ 올바른 예시:
"참가사 150개사 (전년도 기준)"
"2025년 기준 바이어 2,300명 참석"
"2024~2025년 평균 성장률 15%"

❌ 잘못된 예시:
"참가사 150개사" (출처 연도 불명)
"작년 바이어 2,300명" (정확한 연도 누락)
```

### 프로그램/일정
```
✅ 올바른 예시:
"2026년 KIBCON 세미나 (5월 14일)"
"신제품 쇼케이스 - 2026년 트렌드 발표"

❌ 잘못된 예시:
"올해 KIBCON" (연도 불명)
"최신 트렌드 발표" (시점 모호)
```

### 푸터 저작권
```
✅ 권장 형식:
"© 2026 KIBEX. All rights reserved."
또는
"© 2024-2026 KIBEX. All rights reserved." (이력 강조 시)

❌ 지양 형식:
"© 2024 KIBEX" (과거 연도만 표기)
```

## 3) 파일별 적용 기준

### `final-copy/*.md`
- **현재 행사 정보**: 모두 2026
- **실적 데이터**: `전년도 기준` 또는 `2025년 기준` 명시
- **과거 참조**: 필요 시에만 `2024년`, `2023년` 등 구체 연도

### `planning/*.md`
- 기획 단계에서 연도 플레이스홀더 사용 가능 (`{YEAR}`)
- 최종 배포 전 `final-copy`로 이동 시 2026으로 확정

### 프론트엔드 구현
- 하드코딩 금지: 연도를 변수화하여 관리
  ```js
  const CURRENT_YEAR = 2026;
  const LAST_YEAR = 2025;
  ```
- 배포 시점에 설정 파일 또는 환경 변수로 관리 권장

## 4) 연도 전환 프로세스 (2027년 준비 시)

### 4-1. 사전 작업 (행사 종료 후)
1. 2026년 실적 데이터 수집
   - 참가사 수, 방문객 수, 프로그램 참여율 등

2. 아카이브 정리
   - `content/archive/2026/` 폴더 생성
   - 2026년 최종 콘텐츠 백업

### 4-2. 문구 일괄 변경
1. 연도 변수 업데이트
   ```
   CURRENT_YEAR: 2026 → 2027
   LAST_YEAR: 2025 → 2026
   ```

2. `final-copy/*.md` 일괄 검색/치환
   - "2026년 5월" → "2027년 5월"
   - "전년도 기준" 데이터 → 2026년 실적으로 갱신

3. 푸터 저작권 업데이트
   - `© 2024-2026` → `© 2024-2027`

### 4-3. 검증 체크리스트
- [ ] Hero 섹션 연도 확인
- [ ] CTA 버튼 문구 연도 확인
- [ ] 실적 데이터 라벨 확인 (`전년도 기준` 유지)
- [ ] 푸터 저작권 연도 확인
- [ ] 404/에러 페이지 연도 확인
